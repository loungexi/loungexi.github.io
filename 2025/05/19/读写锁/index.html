什么是读写锁？

读写锁是一种在同一时刻允许多个读线程访问，但是在写线程访问时，所有的读线程和读线程均被阻塞的锁。

读写锁维护了一对锁，一个读锁和一个写锁，通过分离读锁和写锁，使得并发性比一般的排他锁提升了很多。Java并发包提供读写锁接口是ReadWriteLock，其实现是ReentrantReadWriteLock，特性如下：

ReentrantReadWriteLock的特性
特性 说明
公平性选择 支持非公平（默认）和公平的锁获取方式，吞吐量是非公平优于公平。
重进入 该锁支持重进入，以读写线程为例：读线程在获取了读锁之后，能够再次获取到读锁。而写线程在获取了写锁之后能够再次获取到写锁，也可以获取读锁。
锁降级 遵循获取写锁 –> 获取读锁 –> 释放写锁的次序，写锁能够降级成为读锁。

ReadWriteLock仅定义了获取读写锁和写锁两个方法，即readLock()方法和writeLock()方法，而它的实现ReentrantReadWriteLock除了接口方法之外还提供了一些便于外界监控内部工作状态的方法：

ReentrantReadWriteLock提供的监控内部工作状态的方法
方法名称 描述
int getReadLockCount() 返回当前读锁被获取的次数。该次数不等于获取读锁的线程数，例如，仅一个线程，它连续获取（重进入）了n次读锁，那么占据读锁的线程数是1，但该方法返回n。
int getReadHoldCount() 返回当前线程获取读锁的次数。
boolean isWriteLocked 判断当前写锁是否被获取
int getWriteHoldCount 返回当前写锁被获取的次数

ReentrantReadWriteLock的实现分析

ReentrantReadWriteLock的实现主要包括：读写状态的设计、写锁的获取与释放、读锁的获取与释放以及锁降级。

读写状态的设计

读写锁依赖自定义同步器（继承AQS）来实现同步功能，读写状态就是其同步器的同步状态。读写锁的自定义同步器需要在同步状态（一个32位整型变量）上维护多个读线程和一个写线程的状态，所以读写状态的设计十分关键。

如果在一个整型变量上维护多种状态，就一定需要“按位切割使用”这个变量。读写锁将变量切分为两个部分，高16位表示读，低16位表示写。

读写锁是图和迅速确定读和写各自的状态的呢？答案是通过位运算。

假设当前同步状态值为S，写状态值为 S & 0x0000FFFF（将高16位全部抹去），读状态等于 S >>> 16（无符号补0右移16位）。当写状态增加1时，等于S+1。当读状态增加1时，等于 S + (1<<16) ，也就是 S + 0x00010000。

写锁的获取和释放

写锁是一个支持重进入的排他锁。如果当前线程已经获取了写锁，则增加写状态。如果当前线程在获取写锁时，读锁已经被获取（读状态不为0）或者该线程不是已经获取写锁的线程，则当前线程进入等待状态。获取写锁的源码如下：

写锁的释放为每次释放均减少写状态，当写状态为0时表示写锁已经被释放，等待的读写线程能够继续访问读写锁，同时前次写线程的修改对后续读写线程可见。

读锁的获取与释放

读锁是一个支持支持重进入的共享锁，它能够被多个线程同时获取，在没有其他写线程访问（或者写状态为0）时，读锁总会被成功地获取，而获取读锁也只是（线程安全地）增加读状态。如果当前线程已经获取到了读锁，则增加读状态。如果当前线程在获取读锁时，写锁已经被其他线程获取，则进入等待状态。读状态是所有线程获取读锁的次数总和，而每个线程各自获取读锁的次数只能选择保存在ThreadLocal中，由线程自身来维护，这使得读锁的实现变得复杂。

读锁的每次释放都是线程安全的，均减少读状态，减少的值为( 1 << 16 )。

锁降级

锁降级是指写锁降级成为读锁。锁降级是指把持住（当前拥有的）写锁，再次获取到读锁，随后释放（先前拥有的）写锁的过程。

锁降级中读锁的获取是必要的，主要是为了保证数据的可见性，如果当前线程不获取读锁而是直接释放写锁，假设此刻另一个线程（线程T）获取了写锁并修改了数据，那么当前线程无法感知到线程T的数据更新。如果当前线程获取读锁，即遵循锁降级的步骤，则线程T将会被阻塞，直到当前线程使用数据并释放读锁之后，线程T才能湖区写锁进行数据更新。ReentrantReadWriteLock不支持锁升级（把持读锁、获取写锁、释放读锁的过程），目的也是保证数据可见性，如果读锁已被多个线程获取，其中任意线程成功获取了写锁并更新了数据，则他的更新对其他获取到读锁的线程是不可见的。
